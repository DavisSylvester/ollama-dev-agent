🟡 Partial (exists but incomplete)

PatternStatusGap / recommendation
Loop / doom-loop detectionPartialWe added tool-thrash detection (per-tool-name counts). Missing the article's per-file edit-count detection to catch a worker rewriting the same file repeatedly.
Error handling & retriesPartialModel calls retry w/ backoff (invokeWithRetry); coder model has fallback. Missing: per-tool retry, and fallback for planner/reviewer models.
Pre-completion checklistPartialLint gate is enforced, but nothing explicitly verifies each acceptance criterion before the worker exits — that's left to the Reviewer's judgment.
Human-in-the-loopPartialPRD approve/reject only. No pause-before-consequential-action, no inline annotation cycle.
Time budgetingPartialStep-budget warnings exist; no wall-clock budget. Directly relevant: the TASK-009 hang burned ~20 min with no time-based cutoff.
Task delegationPartialLangGraph runs tasks in parallel, but a task's Worker is monolithic — no nested sub-agents with clean context for sub-steps.
Skill loading by contextPartialKB category matching (ui/api/db/auth) is similar in spirit, but there's no executable skills system (sequential workflows / advisory skills).

🔴 Gaps (not present)

PatternStatusWhat's missing & why it matters
Context compaction / summarizationGapODA resets context each iteration instead of summarizing. Within one long ReAct iteration, message history can still grow unbounded — no SummarizationMiddleware / ContextEditing. Biggest structural gap for long tasks.
Reasoning budget allocationGapNo "reasoning sandwich" — no deliberate allocation of model thinking toward planning & verification over implementation.
Trace analysis for self-improvementGapNo automated harvesting of run traces to find failure modes and tune the harness. The KB is the closest, but it's populated reactively, not from systematic trace analysis.
Todo-list scratchpad (in-task)GapThe Worker has no explicit todo list to decompose a task and track sub-steps within its own context window.
Annotation cycle / backflowGapOnce the PRD is approved, tasks are fixed. No inline human annotation of plans, and no mechanism for mid-implementation discovery to push work back to re-planning.
Skills architecture (namespaces)GapNo spec: / oracle: / code:-style skills that behave differently (workflow vs advisory vs utility).
Delta tracking (brownfield)GapNo ADDED/MODIFIED/REMOVED marking for evolving specs / existing codebases.
Process/resource hygieneGapSurfaced today: shell_exec can spawn long-lived servers that aren't reliably killed on Windows, leaking processes across runs. No process-tree cleanup or server-start guard.
Highest-leverage gaps to close next
  1. Context compaction inside the ReAct loop (summarize tool output / old turns) — the one structural gap that limits long, hard tasks like TASK-009.
  2. Wall-clock time budget + per-file edit-loop detection — would have auto-aborted today's 20-minute shell_exec hang instead of needing a manual kill.
  3. Process hygiene / server-start guard — kill the shell_exec process tree on timeout; block/guard long-lived listen() commands (KB anti-pattern added today is a stopgap).
  4. Pre-completion checklist — verify every acceptance criterion before exit, not just lint + reviewer judgment.
  5. In-task todo list + sub-agents — let a hard task decompose itself with clean sub-contexts.
